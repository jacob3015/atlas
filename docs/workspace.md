# Atlas Workspace

This document defines the structure and ownership rules of the Atlas workspace.
It is intended both as a setup guide for Atlas users and as a reference for
future development.

## Overview

Atlas stores local input and generated data in the `.atlas` directory of the
workspace.

```text
.atlas/
├── data/
└── cache/
```

The two directories are separated according to who creates and manages the
data:

- `data` contains input data saved directly by the user.
- `cache` contains reusable data generated by Atlas.

The `.atlas` directory is excluded from Git and must not contain source code or
files that need to be shared through the repository.

## Data

`.atlas/data` contains data that the user obtains outside Atlas and saves in the
workspace explicitly.

Examples include CSV files downloaded manually from the KRX Data Marketplace:

```text
.atlas/data/krx
├── etf/
│   └── etf_20260713.csv
└── kospi/
    └── kospi_20260713.csv
```

Atlas treats these files as user-owned input data.

- Atlas may read the files to build derived data.
- Atlas must not delete or overwrite them automatically.
- The user is responsible for downloading, updating, backing up, and removing
  them.
- Removing `.atlas/cache` must not remove anything from `.atlas/data`.

## Cache

`.atlas/cache` contains data generated by Atlas from user-provided data or
external providers.

Examples include:

- Parquet files built from CSV files in `.atlas/data`.
- Parquet files built from KRX OpenAPI responses.

```text
.atlas/cache/
├── etf/
│   └── etf.parquet
├── kospi/
│   └── kospi.parquet
└── krx/
    ├── etf_daily_trading/
    ├── kospi_series_daily_price/
    ├── kospi_stock_basic_info/
    └── kospi_stock_daily_trading/
```

Cache data is derived data and can be regenerated by Atlas. It is used to avoid
unnecessary processing and repeated external API requests.

KRX OpenAPI caches are especially important because each API has a daily request
limit and responses may take a long time. Reusing a cached response prevents the
same base date from consuming the request quota repeatedly and avoids repeated
network delays.

Although cache data is reproducible, Atlas treats it as persistent by default:

- Atlas may create and update cache files as part of an explicit operation.
- Atlas should reuse a valid cache before making the same API request again.
- Atlas must not expire or delete cache files automatically unless an explicit
  cache-retention policy is introduced.
- The user may delete cache files when regeneration is acceptable.

## Classification Rule

Use the following questions when deciding where a file belongs:

1. Was the file obtained outside Atlas and saved directly by the user?
   Store it in `.atlas/data`.
2. Was the file generated by Atlas from input data or a Provider response?
   Store it in `.atlas/cache`.

The ability to regenerate a file is also an important distinction. Deleting a
cache may cost time or API requests, but Atlas must be able to regenerate it.
Deleting user-owned data may require the user to obtain the source file again.

## Examples

| File | Location                 | Owner | Regeneration |
| --- |--------------------------| --- | --- |
| KRX ETF master CSV downloaded by the user | `.atlas/data/krx/etf/`   | User | Download again manually |
| KRX KOSPI constituents CSV downloaded by the user | `.atlas/data/krx/kospi/` | User | Download again manually |
| ETF master Parquet generated from CSV | `.atlas/cache/etf/`      | Atlas | Build from user data |
| KOSPI constituents Parquet generated from CSV | `.atlas/cache/kospi/`    | Atlas | Build from user data |
| KRX API response converted to Parquet | `.atlas/cache/krx/`      | Atlas | Request from the Provider again |

## Operational Notes

- Run Atlas from the intended workspace so that it resolves the correct
  `.atlas` directory.
- Back up `.atlas/data` when its source files are difficult to obtain again.
- Consider the API request quota and response time before clearing
  `.atlas/cache/krx`.
- KRX OpenAPI authentication keys are stored in the operating system credential
  store and do not belong in either `.atlas/data` or `.atlas/cache`.
