# Config, Workspace, and Storage

This document defines the responsibilities and boundaries of the `config`,
`workspace`, and `storage` packages in Atlas. It is both a user-facing overview
and an architectural reference for future development.

## Overview

The three concepts answer different questions:

| Concept | Question | Examples |
| --- | --- | --- |
| Config | How should Atlas run? | Authentication keys, user preferences, timeouts |
| Workspace | Where does Atlas keep local data? | `.atlas/data`, `.atlas/cache` |
| Storage | How does Atlas read and write data? | CSV parsing, Parquet serialization |

Keeping these responsibilities separate prevents configuration, path policy,
and file-format logic from becoming coupled.

## Config

Config manages user-defined settings and credentials required to run Atlas.
Configuration values influence application behavior but do not contain market
or investment data.

Examples include:

- KRX OpenAPI authentication keys.
- Default exchanges, currencies, or timezones.
- API timeouts and request-related options.
- User-selected output preferences.
- A user-selected workspace location, if Atlas supports one in the future.

Authentication keys may be represented by a configuration model, but their
secret values must be stored in the operating system credential store rather
than in the repository or `.atlas` directory.

Config is responsible for:

- Defining configuration models.
- Loading settings required by the application.
- Reading and writing credentials through a secure credential store.
- Validating user-provided configuration values.
- Providing commands for explicit configuration changes.

Config is not responsible for:

- Defining the internal structure of `.atlas`.
- Reading or writing market-data files.
- Calling market-data providers.
- Deciding whether cached data should be reused.

## Workspace

Workspace manages the local data space used by Atlas. It defines where
user-provided data and Atlas-generated data are located and establishes the
ownership rules for those locations.

The default workspace directory is `.atlas`:

```text
.atlas/
├── data/
└── cache/
```

- `.atlas/data` contains data obtained outside Atlas and stored directly by the
  user.
- `.atlas/cache` contains reusable data generated by Atlas from user data or
  Provider responses.

Workspace is responsible for:

- Representing the workspace root.
- Providing paths to the `data` and `cache` directories.
- Applying a consistent directory layout for datasets and providers.
- Enforcing the distinction between user-owned data and Atlas-generated cache.
- Supporting explicit workspace initialization or validation when needed.

Workspace is not responsible for:

- Parsing CSV files.
- Serializing DataFrames as Parquet.
- Understanding dataset columns or schemas.
- Calling external APIs.
- Storing credentials.
- Deleting user-owned data.

For detailed workspace ownership and lifecycle rules, see
[`docs/workspace.md`](../workspace.md).

## Storage

Storage provides the mechanisms used to read and write data at paths supplied
by the Workspace.

Examples include:

- Reading a KRX CSV file with the appropriate character encoding.
- Validating required dataset columns.
- Normalizing ticker values before persistence.
- Writing a DataFrame to Parquet.
- Reading a previously generated Parquet cache.
- Checking whether a specific storage target exists.

Storage is responsible for:

- File-format-specific reading and writing.
- Dataset schema validation when required by an adapter.
- Serialization and deserialization.
- Format-related normalization needed for reliable persistence.
- Reporting missing or invalid storage inputs.

Storage is not responsible for:

- Choosing the workspace root.
- Embedding `.atlas/data` or `.atlas/cache` paths.
- Loading authentication keys.
- Calling external APIs.
- Deciding when data should be fetched or when a cache should be used.
- Applying cache expiration or retention policy.

Storage implementations should receive their file or directory paths through
constructor arguments. This keeps them reusable in tests, temporary directories,
and alternative workspace locations.

## Interaction

The application bootstrap composes the three responsibilities with Providers
and Services:

```text
Config ────────────────> Provider
  supplies credentials     calls an external API

Workspace ─────────────> Storage
  supplies paths            reads and writes data

Provider + Storage ─────> Service
                          decides whether to reuse cached data or fetch it
```

## Placement Guidelines

Use the following questions when deciding where new code belongs.

### Add it to Config when

- It represents a user-controlled application setting.
- It loads, validates, or securely stores configuration or credentials.
- It changes how Atlas behaves rather than where data is stored.

### Add it to Workspace when

- It defines or resolves a path under `.atlas`.
- It describes ownership of user data or generated cache data.
- It initializes or validates workspace directory structure.

### Add it to Storage when

- It reads or writes a particular file format.
- It validates a persisted dataset schema.
- It serializes or deserializes a DataFrame.
- It can operate correctly when given an arbitrary path outside `.atlas`.

### Add it elsewhere when

- It calls an external API: use a Provider.
- It decides whether to call a Provider or reuse a cache: use a Service.
- It defines static market metadata: use a Registry.
- It implements CLI interaction: use a Command module.

## Design Rules

The following rules should remain stable as Atlas evolves:

1. Config must not store credentials in `.atlas`.
2. Workspace must distinguish user-owned data from Atlas-generated cache.
3. Storage must not hard-code workspace paths.
4. Providers must not write cache files directly.
5. Services coordinate Providers and Storage when cache behavior is required.
6. Atlas must not automatically delete or overwrite user-owned data.
7. Cache may be regenerated, but API limits and response times should be
   considered before deletion or refresh.
